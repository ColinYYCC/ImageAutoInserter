# TRAE AI 编程项目规则

## 第 1 层：核心原则（不可动摇）

### 原则 1：文档先行
> "没有文档就不写代码。文档是思考的外化，是 AI 理解意图的桥梁。" — Kent Beck

- 每个开发任务必须先创建对应的文档
- 代码变更必须同步更新相关文档
- 文档必须包含：需求边界、验收标准、设计思路

### 原则 2：质量门控
> "每个阶段必须有明确的验收标准，未通过不得进入下一阶段。这是防止技术债务积累的关键。" — Martin Fowler

- 每个工作流阶段都有明确的检查清单
- 必须逐项确认通过后才能进入下一阶段
- 质量门控检查结果必须记录在文档中

### 原则 3：原子任务
> "任务必须拆到 2-5 分钟能完成的粒度。AI 在小任务上表现最好，这是充分利用 AI 能力的关键。" — Andrej Karpathy

- 单个任务应在 2-5 分钟内完成
- 每个任务必须可独立验证
- 任务之间必须无循环依赖

### 原则 4：测试优先
> "RED → GREEN → REFACTOR，这个顺序不能变。必须亲眼看到测试失败，这是保证测试有效性的唯一方法。" — Kent Beck

- 必须先写测试，再写实现代码
- 必须亲眼看到测试失败（RED 阶段）
- 写最少代码让测试通过（GREEN 阶段）
- 优化代码但保持测试通过（REFACTOR 阶段）

### 原则 5：及时中断
> "遇到不确定立即停止。连续 3 次失败必须质疑方案，不要死磕。" — Jeff Dean

- 遇到无法自主决策的问题立即停止
- 连续 3 次修复尝试失败必须中断
- 技术实现出现阻塞立即寻求人工确认

### 原则 6：缺陷修复同样需要文档
> "任何代码修改都应被记录，无论大小。" — 项目规则

- 发现 bug/缺陷后，必须**立即**调用 `Skill: self-improvement` 记录到 `.learnings/ERRORS.md`
- 分析根因、制定修复方案后再动手
- 修复完成后再次调用 `Skill: self-improvement` 记录解决经验到 `.learnings/LEARNINGS.md`
- 禁止：无记录直接修改代码

### 原则 7：范围控制
> "指定修改仅限该部分，绝不超范围变动。" — 项目规则

- 用户指定的修改范围是绝对边界，禁止扩展
- 发现范围外问题时，必须先报告确认，不得自行修改
- 禁止"顺手优化"、"顺便重构"等行为
- 修改完成后立即停止，不得继续其他改动

## 第 2 层：技术约束

### 安全规范
- 禁止将 API 密钥、密码等敏感信息硬编码在代码中
- 必须使用`.env`文件管理敏感配置
- 必须确保`.env`文件已添加到`.gitignore`
- 只能通过环境变量读取敏感信息

### 代码规范
- 函数命名使用动词开头（`get_`, `calculate_`, `validate_`）
- 类名使用大驼峰命名法（`UserService`, `DataProcessor`）
- 变量名使用小驼峰或蛇形命名法（根据语言习惯统一）
- 必须为所有函数添加函数级注释（功能描述、参数说明、返回值）

### 注释规范示例
```python
def calculate_total_price(items: list, discount: float = 0.0) -> float:
    """计算订单总价格"""
    # 参数：items (商品列表), discount (折扣率 0.0-1.0)
    # 返回：总价格（float）
```

## 第 3 层：工作流规范（6A 工作流）

### A1: Align（对齐）- 需求澄清

必须执行:
1. 创建 `ALIGNMENT_[任务名].md` 文档
2. 分析项目上下文（结构/技术栈/约束）
3. 明确需求边界（包含什么、不包含什么）
4. 识别并澄清所有歧义点
5. 生成 `CONSENSUS_[任务名].md` 共识文档

质量门控检查:
- [ ] 需求边界清晰无歧义
- [ ] 技术方案与现有架构对齐
- [ ] 验收标准具体可测试
- [ ] 所有关键假设已确认

---

### A2: Architect（架构）- 系统设计

必须执行:
1. 基于 `CONSENSUS` 文档设计系统架构
2. 创建 `DESIGN_[任务名].md` 文档，包含：
   - 架构图（使用 Mermaid）
   - 分层设计和核心组件
   - 接口契约定义
   - 数据流向图
   - 异常处理策略
3. 区分可逆决策和不可逆决策

质量门控检查:
- [ ] 架构图清晰准确
- [ ] 接口定义完整
- [ ] 与现有系统无冲突
- [ ] 设计可行性已验证

---

### A3: Atomize（原子化）- 任务拆分

必须执行:
1. 基于 `DESIGN` 文档创建 `TASK_[任务名].md`
2. 每个任务必须定义：
   - 输入契约（前置条件/数据/环境）
   - 输出契约（交付物/验收标准）
   - 实现约束（技术栈/规范）
   - 依赖关系（前置/并行任务）
3. 生成任务依赖图（Mermaid DAG）

质量门控检查:
- [ ] 任务覆盖完整需求
- [ ] 依赖关系无循环
- [ ] 每个任务可独立验证
- [ ] 复杂度评估合理（2-5 分钟完成）

---

### A4: Approve（审批）- 人工审查

必须执行:
1. 完整性检查：覆盖所有需求
2. 一致性检查：与前期文档一致
3. 可行性检查：技术方案可实现
4. 可控性检查：风险可接受
5. 可测性检查：验收标准明确

审批清单:
- [ ] 需求实现路径清晰
- [ ] 子任务定义明确
- [ ] 边界和限制已标注
- [ ] 验收标准可量化
- [ ] 代码/测试/文档质量标准明确

---

### A5: Automate（自动化）- TDD 执行

TDD 循环（必须严格执行）:

1. RED: 编写失败测试
   - 写一个测试，描述预期功能
   - 运行测试 → 必须看到失败
   - 禁止跳过这一步！

2. GREEN: 编写最少代码
   - 写最少的代码让测试通过
   - 不要过度实现
   - 运行测试 → 确认通过

3. REFACTOR: 优化代码
   - 改进命名、移除重复、简化逻辑
   - 运行测试 → 保持绿色

常见错误（禁止）:
- ❌ 跳过 RED 验证（必须亲眼看到测试失败）
- ❌ 测试命名模糊（使用 `test_功能_场景_预期` 格式）
- ❌ 过度实现（只写最少代码）
- ❌ 测试间依赖（每个测试独立设置环境）

执行前检查:
- [ ] 输入契约满足
- [ ] 环境准备完成
- [ ] 依赖任务已完成

执行后检查:
- [ ] 单元测试通过（边界/异常）
- [ ] 相关文档已更新
- [ ] 代码符合项目规范

---

### A6: Assess（评估）- 质量验收

必须执行:
1. 更新 `ACCEPTANCE_[任务名].md`
2. 整体验收检查：
   - 所有需求已实现
   - 验收标准全部满足
   - 项目编译通过
   - 所有测试通过
3. 质量评估：
   - 代码质量（规范/可读性/复杂度）
   - 测试质量（覆盖率/用例有效性）
   - 文档质量（完整性/准确性）
4. 生成 `FINAL_[任务名].md`（项目总结）
5. 生成 `TODO_[任务名].md`（待办事项）

## 第 4 层：质量门控速查

| 阶段 | 检查项 | 关键问题 |
|------|--------|----------|
| Align | 需求边界清晰 | "不包含什么？" |
| | 技术方案对齐 | "与现有架构冲突吗？" |
| | 验收标准可测 | "如何验证完成？" |
| Architect | 架构图清晰 | "新开发者能看懂吗？" |
| | 接口完整 | "输入输出定义了吗？" |
| | 无冲突 | "与现有代码兼容吗？" |
| Atomize | 覆盖完整 | "所有需求都有任务吗？" |
| | 依赖无环 | "任务能按序执行吗？" |
| | 可独立验证 | "能单独测试吗？" |
| Automate | TDD 流程 | "先测试后实现了吗？" |
| | 代码规范 | "符合项目风格吗？" |
| | 文档同步 | "代码变文档更新了吗？" |
| | 范围控制 | "只修改了指定部分吗？" |
| Assess | 测试通过 | "100% 通过了吗？" |
| | 需求实现 | "所有需求完成了吗？" |
| | 无技术债务 | "有临时方案吗？" |

## 文档模板

### 开发任务文档
```
docs/<任务名>/
├── ALIGNMENT_<任务名>.md    # 需求对齐
├── CONSENSUS_<任务名>.md    # 共识确认
├── DESIGN_<任务名>.md       # 架构设计
├── TASK_<任务名>.md         # 任务拆分
├── ACCEPTANCE_<任务名>.md   # 验收记录
├── FINAL_<任务名>.md        # 项目总结
└── TODO_<任务名>.md         # 待办事项
```

### 缺陷修复文档
```
docs/ALIGNMENT_<问题名>.md  # 问题描述、根因分析、修复方案
```

## 中断条件（立即停止并报告）

遇到以下情况必须立即停止并寻求人工确认：
- 🛑 遇到无法自主决策的问题
- 🛑 技术实现出现阻塞
- 🛑 文档不一致需要确认
- 🛑 连续 3 次修复尝试失败
- 🛑 发现范围外需要修改的问题
- 🛑 有冲动进行"顺便优化"或"顺手重构"
